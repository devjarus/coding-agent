# v2.1 Improvement Plan

Forward-looking doc. Not historical. Captures concrete improvements grounded in two feedback streams from real sessions, filtered against the plugin's "minimum ceremony" principle.

## Source feedback

**Stream 1 — operational pain (10 items).** Day-2 ops without a playbook, CI green / prod broken (CORS), env-var drift undetected for weeks, micro-vs-small misclassification, lost open threads after `/compact`, etc.

**Stream 2 — debugger underuse.** User said "delete throws 500." Orchestrator skipped debugger dispatch, diagnosed inline, patched on a guess. Diagnosis stayed in transcript instead of becoming a `diagnosis.md` artifact. Got lucky.

Common root cause across both: **the pipeline assumes the user already knows what's broken and what should change.** When they don't, the orchestrator shortcuts the discipline.

## Guiding principle

Per `~/.claude/projects/.../memory/MEMORY.md` and the architecture decisions in `AGENTS.md`: **prefer prompt edits over enforcement hooks; add a check or protocol only when prompt fixes fail.** Every new file is baseline context every session forever.

The bar for "is this worth adding?":
- Does it solve pain that hit twice?
- Is it irreducible to a prompt edit?
- Does it survive compaction without re-explaining?

## Changes that pass the bar

### 1. Diagnose-first rule — orchestrator prompt edit

**File:** `agents/orchestrator.md` § Task classification

Add to the size classifier:

> **Bug reports.** If the user message describes a symptom without a cause ("throws 500", "doesn't work", "missing", "broken"), dispatch `debugger` BEFORE classifying size. Implementor only gets dispatched after `diagnosis.md` (or inspection note) exists. Skip only when the user has already named the file + line + cause.

**Solves:** debugger underuse, diagnoses-as-transcript instead of artifact.
**Cost:** ~6 lines.

### 2. Sharper micro classifier — orchestrator prompt edit

**File:** `agents/orchestrator.md` § Size table

Replace current `micro: ≤30 lines, mechanical, no new logic` with:

> **micro** — ≤2 files, additive only, no LOAD-BEARING markers near edit, has clear test target. Orchestrator writes inline.

**Solves:** small touch-ups dispatched as full implementor work.
**Cost:** 1 line.

### 3. `open-threads.md` — new artifact

**Files:**
- new `templates/open-threads.template.md`
- `agents/orchestrator.md` § Read-on-session-start table — add row
- `agents/orchestrator.md` — add update rule

```markdown
---
artifact: open-threads
writer: orchestrator
mutability: append-only
---

## Open Threads
<!-- Append-only. Lines removed only when explicitly resolved. -->
- [2026-05-03] Linear UUID setup pending — user said "later this week"
- [2026-05-03] CORS smoke check missing for /api/feedback DELETE
```

**Solves:** compaction loses user-blocking items. Survives `/clear` and `/compact`.
**Cost:** 1 template + 2 prompt rules.

### 4. Deployment structure (mirrors coding side)

The pipeline ends at `git push`. Deployment is an unmodeled void. Add minimum primitives.

#### 4a. `deployments.md` — append-only log

**File:** `.coding-agent/deployments.md` (per project), template at `templates/deployments.template.md`

```markdown
## 2026-05-03 — production
- commit: a1b2c3d
- preflight: env-vars-present ✓, pre-smoke ✓
- execute: railway up (exit 0)
- verify: smoke ✓ (3/3 URLs)
- status: deployed
```

**Solves:** "what's running in prod after compaction?" — read on session start.

#### 4b. `environments.md` — current state

**File:** `.coding-agent/environments.md` (per project), template at `templates/environments.template.md`

```markdown
## production
- platform: railway
- commit_running: a1b2c3d
- last_verified: 2026-05-03T14:22:00Z
- expected_env_vars: [DATABASE_URL, PUBLIC_WEB_BASE_URL, JWT_SECRET]
- deploy_command: railway up --service api
- env_list_command: railway variables --service api --json
- verify_urls:
  - https://app.example.com/api/health
```

**Solves:** project declares HOW to deploy/verify; plugin runs whatever's declared. No platform adapters.

#### 4c. Deploy mode in classifier

**File:** `agents/orchestrator.md` § Task classification

Add ~15 lines as a prompt rule (no separate protocol file):

```markdown
### Deploy mode

Triggered when user says "deploy", "ship", "push to prod", "rollback", "bump env".

1. Intake — classify: deploy / rollback / env-change. Identify target env.
2. Preflight — read environments.md. Run env_list_command, diff vs expected_env_vars.
   Run pre-smoke if defined. ABORT and surface to user if either fails.
3. Execute — run deploy_command. Capture exit + tail.
4. Verify — hit verify_urls. On failure: append to open-threads.md, surface, do NOT mark deployed.
5. Record — append to deployments.md, update environments.md commit_running + last_verified.
```

#### 4d. One new check

**File:** `checks/env-vars-present.sh`

Reads expected vars from environments.md, runs the project's `env_list_command`, diffs. Project-agnostic because the command itself is declared per-project.

**Solves:** env-var drift caught before deploy, not weeks later.
**Cost (4a–4d):** 2 templates + 1 check + ~15 prompt lines.

### 5. AGENTS.md vendor-neutrality enforcement

The plugin updates AGENTS.md during close-out. Today there's no explicit rule preventing plugin-specific bleed (deploy commands, plugin paths, etc.). The agents.md community spec is now widely adopted; the plugin should respect it.

**Files:**
- `skills/practices/project-docs/SKILL.md` — add ~10-line "AGENTS.md is vendor-neutral" section
- `protocols/close-out.md` step 3 — add 1-line reminder

```markdown
## AGENTS.md is vendor-neutral

AGENTS.md follows the agents.md community spec (https://agents.md). It must be
useful to ANY coding agent: Cursor, Aider, Codex, Claude Code with or without
this plugin.

❌ No references to .coding-agent/, protocols, checks, or skills by name
❌ No coding-agent: prefixed instructions
❌ No state, deploy commands, or env tracking config
✓ Stack, build/test commands, conventions, architecture, known gotchas

Plugin-specific configuration lives in .coding-agent/. The plugin reads its own
files; it doesn't need AGENTS.md to point at them.
```

**Solves:** users can rip out the plugin and AGENTS.md still works for whatever agent they switch to. No vendor lock-in.
**Cost:** ~11 lines across two files.

## Implementation order

Order by leverage × independence — each phase shippable in one session, validates with `./scripts/validate.sh`.

### Phase 1 — Prompt-only (~30 min)

- Diagnose-first rule (#1)
- Sharper micro classifier (#2)
- AGENTS.md vendor-neutrality rule in `project-docs` skill + `close-out` protocol (#5)

No new files. Pure orchestrator and skill prompt edits. Lowest risk.

### Phase 2 — One-artifact wins (~30 min)

- `open-threads.md` template + orchestrator wiring (#3)

One template, two prompt rules. Solves compaction-survival.

### Phase 3 — Deployment primitives (~1.5 hr)

- `deployments.md`, `environments.md` templates (#4a, #4b)
- Deploy mode in orchestrator classifier (#4c)
- `env-vars-present.sh` check (#4d)

Largest cohesive change. Defer until phases 1–2 land and bake.

## What this does not address

Honest list of pain that's known but deferred:

- **Test-as-deployed coverage gaps.** The CORS bug shipped because integration tests use `app.inject()`, bypassing the real proxy. Plugin-level fix would require detecting Next.js + proxied API patterns. Defer until a second instance.
- **Compaction itself.** Phase 2 helps survival but doesn't prevent compaction. Action-log compaction (already shipped in v2.0.x) plus open-threads is the current answer.
- **Env-var values, not just names.** `env-vars-present` checks names only. Detecting "value is wrong" needs runtime introspection, not feasible plugin-side.
- **Multi-environment coordination.** environments.md handles each env independently. Cross-env workflows (e.g., "deploy to staging, smoke, then prod") need orchestration not modeled here. Defer.

These are tracked here so they're not forgotten, not because they're scheduled.
